# AIエージェント設計思想
**Naomi's Agent Design Philosophy v1.0**

> 「Claudeは育つ。設計は腐る。だから設計も育てる。」

このドキュメントは、AIエージェント開発のすべてのプロジェクトで  
設計・要件定義フェーズに通読し、思想として組み込むためのものです。

開発中に都度参照するリファレンスではありません。  
**設計前に読んで、判断軸を自分の中に入れる**ために使います。

---

## なぜこの思想が必要か

AIエージェントは「動けばいい」で作ると、必ず後で詰まります。

理由は2つあります。

**理由①：Claudeはどんどん賢くなる**

3ヶ月前に「Claudeにはできないからアプリ側で補う」と書いたコードが、  
今のClaudeには不要になっていることがある。  
でも誰もそのコードを消さない。  
結果として「動いているが、無駄が多く、モデル更新のたびに壊れる」システムになる。

**理由②：雑に作っても一応動く**

これがいちばん厄介です。  
AIエージェントはハーネス（Claude周りの制御コード）がぐちゃぐちゃでも、  
プロンプトに全部詰め込んでも、とりあえず動く。  
「遅い・コスト高い・壊れやすい」という痛みを経験するまで、  
設計を見直す動機が生まれにくい。

だから**設計段階でこの思想を組み込む**必要があります。

---

## ハーネスとは何か

まずこの言葉を理解することが出発点です。

**ハーネス = Claude周りのアプリ側コード全部**

具体的には：
- ツール（検索・ファイル読み書き・API呼び出しなど）の定義
- システムプロンプトの内容
- ツール結果をClaudeに渡す前の整形処理
- Claudeの出力を次の処理に渡す変換処理
- エラーハンドリング・リトライ処理

これらは「必要だから書いた」という確信のもとで作られますが、  
その確信は古くなります。  
**書き出して、一個ずつ「本当に必要か？」と問い直すことが設計の起点です。**

## 5つの判断軸

### 判断軸①　ツールはbashで十分か、専用ツールにすべきか

**考え方**

bashはなんでもできる。これが問題です。

Claudeにbashツールだけ渡すと、安全な操作も危険な操作も  
「コマンド文字列」という同じ形でやってきます。  
アプリ側はそれが危険かどうか判断できません。

**判断基準：以下のいずれかに当てはまる → 専用ツールにする**

| 条件 | 理由 |
|---|---|
| 取り消せない操作（送信・削除・上書き） | ユーザー確認を挟めるようにする |
| ユーザーへの確認が必要な操作 | ダイアログ表示できるようにする |
| ログ・監査が必要な操作 | 型付き引数としてログに残せるようにする |

**悪い例（bash一本）**
```bash
curl -X POST api/send-email   # メール送信（取り消せない）
rm -rf ./data/old             # データ削除（取り消せない）
cat report.txt                # ファイル読み込み（安全）
```
→ アプリ側には全部「コマンド文字列」として届く。危険な操作を止められない。

**良い例（専用ツール化）**
```
send_email(to, subject, body)     # 専用ツール → 確認ダイアログを挟める
delete_file(path)                 # 専用ツール → 確認ダイアログを挟める
read_file(path)                   # bashのままでOK
```

**マルチエージェントでの注意点**

複数エージェントが同じリソースにアクセスする場合、  
どのエージェントが何の権限を持つかを設計段階で明確にする。  
「どれかのエージェントがやってくれるだろう」は事故の元。

---

### 判断軸②　静的な内容と動的な内容を分けているか

**考え方**

Claudeへのリクエストは毎回「全部送り直し」が基本です。  
ただしAnthropicのAPIには**キャッシュ**があり、  
前回と同じ部分はキャッシュ代（通常の10%）で済みます。

キャッシュが効くかどうかは**順番**で決まります。

| 内容 | 変わる頻度 | 置く場所 |
|---|---|---|
| ツール定義・システムプロンプト | ほぼ変わらない＝静的 | **前に置く** |
| 会話履歴・ツール結果 | 毎回変わる＝動的 | **後に置く** |

**悪い例**
```
[動的：今回のユーザー入力]
[静的：ツール定義]
[静的：システムプロンプト]
```
→ 先頭が毎回変わるのでキャッシュが一切効かない。毎回フルコスト。

**良い例**
```
[静的：システムプロンプト]
[静的：ツール定義]
[動的：会話履歴・ツール結果]
[動的：今回のユーザー入力]
```
→ 前半がキャッシュにヒットし続ける。コストが大幅に下がる。

**他にキャッシュを壊すNG行動**
- セッション中にモデルを切り替える（キャッシュはモデル別）
- ツールを途中で追加・削除する（前半が変わるのでキャッシュが壊れる）
- システムプロンプトを毎回書き換える

### 判断軸③　システムプロンプトに全部書いていないか

**考え方**

Claudeはコンテキストウィンドウ全体に「注意」を払います。  
今のタスクに無関係な内容にも処理コストが発生します。  
プロンプトが太るほど、速度が落ち、精度も下がります。

**悪い例（全部詰め込み）**
```
システムプロンプト（3000トークン）
- SNS投稿の書き方ルール
- メール作成のルール
- 競合リサーチの手順
- レポート整形のルール
- ペルソナ設定
- トーン＆マナー
...
```
→ 今「競合リサーチ」しかしないのに、SNS投稿ルールも読み込んでいる。

**良い例（オンデマンド読み込み）**
```
システムプロンプト（200トークン）
- 使えるスキルファイルの一覧と概要だけ書く

→ Claudeが「競合リサーチが必要」と判断したとき
  だけ該当スキルファイルを読みに行く
```

**マルチエージェントでの応用**

各エージェントのシステムプロンプトには  
「そのエージェントの役割」と「使えるスキルの一覧」だけ書く。  
詳細な手順はスキルファイルに分離して、必要なときだけ読ませる。

### 判断軸④　コンテキストの管理をClaudeに任せているか

**考え方**

長期間動くエージェントは、コンテキストウィンドウの上限に近づきます。  
昔は「どこでリセットするか」をアプリ側が管理していました。  
今のClaudeは自分で管理できます。

**任せられること**
- **コンパクション**：過去の会話を自分で要約して継続する
- **メモリフォルダ**：重要な情報を自分でファイルに書き出して後で読む
- **サブエージェント**：特定のタスクを別の新鮮なコンテキストで処理する

**任せる前にやること**

設計段階で「このエージェントはどれくらいの長さのタスクをこなすか」を定義する。  
短期タスク（数ターン）なら管理不要。  
長期タスク（数十〜数百ターン）ならコンパクションかメモリフォルダを設計に入れる。

**マルチエージェントでの注意点**

オーケストレーター（指示を出す側）とワーカー（実行する側）を分けて設計する。  
ワーカーは毎回新鮮なコンテキストで動くことが多いため、  
「ワーカーに何を渡すか」の設計が品質を左右する。

### 判断軸⑤　設計の前提を書き残しているか

**考え方**

Claudeは数ヶ月ごとに能力が変わります。  
設計当初の「Claudeにはできないからアプリ側で補う」という前提が、  
モデル更新後には不要になっていることがある。  
前提を書き残さないと、後で「なぜこのコードがあるのか」が分からなくなります。

**悪い例（注釈なし）**
```python
# 検索結果を200文字に切り詰める
result = result[:200]
```
→ 3ヶ月後に「なんでこれあるんだっけ？」で止まる。

**良い例（前提を明記）**
```python
# Claude 3.5時代、長い結果を渡すと途中で止まる問題があったため
# 現在のモデルでは不要かもしれない → モデル更新時に要確認
result = result[:200]
```
→ モデル更新時に「削っていいか」を即判断できる。

**仕様書への書き方**

ハーネスの制御ロジックを書くとき、以下をセットで記載する。

```
【処理内容】検索結果を200文字に切り詰める
【理由】Claude 3.5では長文入力で停止する問題があったため
【見直しトリガー】モデルをSonnet 4以降に更新したとき
```

## 失敗パターン集

設計段階で避けるべき、よくある間違いをまとめます。

### 失敗①　プロンプト肥大化

**症状**：レスポンスが遅い、コストが高い、指示が守られない箇所がある  
**原因**：システムプロンプトに全タスクの手順を詰め込んでいる  
**対策**：スキルファイルに分離してオンデマンド読み込みに切り替える

### 失敗②　キャッシュが全く効いていない

**症状**：同じエージェントを何度呼んでもコストが変わらない  
**原因**：動的な内容（会話履歴など）が前に来ている、またはセッション中にモデルを切り替えている  
**対策**：静的→動的の順番に並べ替える。モデルはセッション中に変えない。

### 失敗③　bash一本でなんでもやらせている

**症状**：意図しない操作が実行されても気づけない、ログが取れない  
**原因**：専用ツールを作るのが手間で、全部bashに任せている  
**対策**：「取り消せない操作」だけでも専用ツール化する

### 失敗④　エージェントの役割が曖昧

**症状**：同じ処理を複数のエージェントがやっている、誰も処理しない抜け漏れが出る  
**原因**：マルチエージェント設計でオーケストレーターとワーカーの境界が曖昧  
**対策**：設計段階で「このエージェントだけがやること」を一文で書く

### 失敗⑤　モデル更新で突然壊れる

**症状**：新しいモデルに切り替えたら動かなくなった  
**原因**：古いモデルの弱点を補うコードが残っていて、新しいモデルと干渉している  
**対策**：前提を注釈で書き残し、モデル更新時の見直しリストを作っておく

## マルチエージェント固有の設計視点

単一エージェントの設計論だけでは不十分な観点をまとめます。

**エージェント間の受け渡しデータを定義する**

「前のエージェントが出した結果を次のエージェントがそのまま受け取る」設計は、  
フォーマットのズレで簡単に壊れます。  
受け渡しデータの型・構造・必須項目を仕様書に明記する。

**オーケストレーターとワーカーを分ける**

- オーケストレーター：全体の流れを管理・判断する
- ワーカー：特定のタスクだけを実行する

この役割を混在させると、どのエージェントが何を判断しているか分からなくなる。

**エージェントをいつ増やすか**

エージェントを増やすのは以下のときだけにする。
- タスクが並列で実行できる
- タスクが独立していて、お互いの結果を待つ必要がない
- コンテキストを分離した方が精度が上がる

「機能ごとにエージェントを作る」は過剰設計になりやすい。

## 仕様書への落とし込みテンプレート

設計段階でこの形式で書いておくと、後の見直しが楽になります。

### エージェント定義テンプレート

```
【エージェント名】競合リサーチエージェント
【役割】競合他社のWebサイトを調査し、構造化されたレポートを出力する
【使うツール】
  - bash（Web検索・ファイル読み書き）
  - export_report()（専用ツール：レポートの外部保存）※取り消せないため専用化
【読み込むスキル】competitor-research.md（オンデマンド）
【受け取るデータ】調査対象の企業名リスト（JSON配列）
【渡すデータ】競合レポート（Markdown形式）
【見直しトリガー】モデル更新時、月次コストレビュー時
```

### ツール分類テンプレート

```
【ツール名】send_newsletter
【分類】専用ツール
【理由】メール送信は取り消せないため。ユーザー確認ダイアログを挟む。
【前提】Claude 3.x時代からの設計。現在も専用ツールが適切。
```

## 設計レビューチェックリスト

仕様書・要件定義を書いたら、以下で確認します。

### ハーネスの棚卸し
- [ ] アプリ側でやっていることを全部リストアップしたか
- [ ] 各項目に「なぜClaudeに任せないのか」の理由を書いたか
- [ ] その理由は今のClaudeでも有効か確認したか

### ツール設計
- [ ] 取り消せない操作を洗い出したか
- [ ] ユーザー確認が必要な操作を洗い出したか
- [ ] ログ・監査が必要な操作を洗い出したか
- [ ] 上記3つに該当するものを専用ツールにしたか

### コンテキスト設計
- [ ] 静的な内容（ツール定義・システムプロンプト）を前に置いているか
- [ ] システムプロンプトに全タスクの手順を詰め込んでいないか
- [ ] スキルファイルに分離してオンデマンド読み込みにしているか
- [ ] エージェントごとに「何を事前ロードして、何をオンデマンドにするか」を定義したか

### マルチエージェント設計
- [ ] オーケストレーターとワーカーの役割を分けて定義したか
- [ ] エージェント間の受け渡しデータの型・構造を定義したか
- [ ] 各エージェントが「そのエージェントだけがやること」を一文で書けるか

### 陳腐化への備え
- [ ] 制御ロジックに「なぜこれが必要か」の注釈を書いたか
- [ ] 「モデル更新時に見直すリスト」を仕様書に作ったか
